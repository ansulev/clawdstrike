Create `spl-parser.ts` with these exported types and functions:

**Types:**
```typescript
export interface SplCommand {
  command: string;       // "search", "where", "stats", "eval", "table", "fields", "sort", "head", "dedup", "rename", "rex", "lookup", etc.
  rawText: string;       // The original text of this pipe segment
  args: string;          // Everything after the command keyword
}

export interface SplFieldCondition {
  field: string;         // e.g. "CommandLine", "process", "src_ip"
  operator: string;      // "=", "!=", "LIKE", "IN", "match"
  value: string;         // The comparison value (without surrounding quotes)
  modifier: string | null; // "contains" | "startswith" | "endswith" | "regex" | null (derived from LIKE patterns / match())
  negated: boolean;      // true if preceded by NOT
}
```

**Functions:**

1. `parseSplPipeChain(source: string): SplCommand[]` -- Split SPL text into pipe-delimited commands.
   - Strip leading comment lines (lines starting with `//` or `#`).
   - Split on `|` that is NOT inside quotes.
   - For the first segment, if it starts with `search` keyword, set command to "search"; otherwise if it starts with `index=` or `sourcetype=`, treat it as an implicit "search" command.
   - For subsequent segments, the first word is the command name, the rest is args.
   - Handle multi-line SPL (join lines, preserve within-quote content).

2. `parseSplFieldConditions(source: string): SplFieldCondition[]` -- Extract field=value conditions from SPL text.
   - Parse `field="value"`, `field=value` (unquoted), `field="*pattern*"` (LIKE contains), `field="pattern*"` (LIKE startswith), `field="*pattern"` (LIKE endswith).
   - Parse `match(field, "regex")` as regex modifier.
   - Parse `where field LIKE "%value%"` / `where field="value"` forms.
   - Handle `NOT` prefix for negation.
   - Handle `IN ("val1", "val2")` by expanding to multiple conditions with the same field.
   - Return array of all extracted conditions.

3. `matchSplConditions(payload: Record<string, unknown>, conditions: SplFieldCondition[]): { matched: boolean; matchedFields: Array<{ field: string; value: string; matched: boolean }> }` -- Client-side matching of evidence payload against SPL conditions.
   - For each condition, look up `payload[condition.field]` (case-insensitive key lookup).
   - Apply modifier logic: null = exact match (case-insensitive), "contains" = string.includes, "startswith" = string.startsWith, "endswith" = string.endsWith, "regex" = new RegExp test.
   - If `negated`, invert the result.
   - ALL conditions must match for overall `matched: true` (AND logic, matching Sigma adapter behavior).
   - Return per-field match details for explainability trace.

4. `buildSplFromSeed(seed: DraftSeed, fieldMapper: (sigmaField: string) => string | null): string` -- Generate SPL text from a DraftSeed.
   - Map `seed.dataSourceHints` to appropriate `index=` and `sourcetype=` values:
     - "process" / "command" -> `index=main sourcetype=WinEventLog:Security`
     - "file" -> `index=main sourcetype=WinEventLog:Sysmon`
     - "network" -> `index=main sourcetype=firewall`
     - "registry" -> `index=main sourcetype=WinEventLog:Sysmon`
     - default -> `index=main sourcetype=WinEventLog:Security`
   - Map `seed.extractedFields` to CIM field names using the provided `fieldMapper` function.
   - Apply modifiers: use `field="*value*"` for contains semantics.
   - Build pipe chain: `index=... sourcetype=... | where <conditions> | table <fields>`.
   - Add comment header with title and technique hints.

5. `findSplSourceLineHints(source: string, fields: string[]): number[]` -- Find line numbers containing the given field names (for editor highlighting).

**Implementation notes:**
- DO NOT use a full parser combinator library. Use regex-based extraction matching the complexity level of `extractSigmaSelectionFields()` in sigma-adapter.ts.
- The parser is for client-side approximate matching, not full SPL execution. Edge cases (subsearch, macros, eval expressions) are acceptable to miss.
- Export everything from the module (types and functions).

Create `spl-adapter.ts` implementing `DetectionWorkflowAdapter` for `splunk_spl` file type. Follow the sigma-adapter.ts structure closely.

**File type registration (at module load):**
```typescript
registerFileType({
  id: "splunk_spl",
  label: "Splunk SPL Rule",
  shortLabel: "SPL",
  extensions: [".spl"],
  iconColor: "#65a637",
  defaultContent: SPL_DEFAULT_CONTENT,  // see below
  testable: true,
  convertibleTo: ["sigma_rule"],
  detect: (filename: string, content: string) => {
    // Content-based detection: pipe chains, index=, sourcetype=
    if (filename.endsWith(".spl")) return true;
    const lower = content.toLowerCase();
    return (
      (lower.includes("index=") && lower.includes("sourcetype=")) ||
      (lower.includes("| where") && lower.includes("| stats")) ||
      (lower.includes("| search") && lower.includes("|"))
    );
  },
});
```

**SPL_DEFAULT_CONTENT:**
```
// Untitled SPL Detection Rule
// Author: Detection Lab
// Description: Detects suspicious activity
index=main sourcetype=WinEventLog:Security EventCode=4688
| where process="*suspicious*"
| table _time, ComputerName, process, CommandLine
```

**Adapter implementation:**

1. `fileType: "splunk_spl"`

2. `canDraftFrom(seed)` -- Return true when `seed.dataSourceHints` includes any of "process", "command", "file", "network", "registry", or when `seed.preferredFormats` includes "splunk_spl".

3. `buildDraft(seed)` -- Use `buildSplFromSeed()` from spl-parser.ts with `(sigmaField) => translateField(sigmaField, "splunkCIM")` as the field mapper. Set fileType to "splunk_spl", name to `Detection: ${seed.kind} ${seed.id.slice(0, 8)}`, techniqueHints from seed.

4. `buildStarterEvidence(seed, document)` -- Follow sigma-adapter.ts pattern exactly:
   - Create positive dataset items from `seed.sourceEventIds` using `seed.extractedFields`.
   - Map field names to CIM equivalents using `translateField(field, "splunkCIM")` for each key in the payload.
   - Add one negative baseline item `{ baseline: true, process: "benign.exe", ComputerName: "WORKSTATION01" }`.
   - Return EvidencePack with fileType "splunk_spl".

5. `runLab(request)` -- Client-side simulated execution:
   - Get SPL source from `request.adapterRunConfig?.["splSource"]` (same pattern as sigma adapter uses "sigmaSource").
   - If no source, return empty run.
   - Collect all evidence items across datasets (same as sigma adapter).
   - Parse SPL conditions using `parseSplFieldConditions(source)`.
   - For each structured_event/ocsf_event evidence item, run `matchSplConditions(item.payload, conditions)`.
   - Build `plugin_trace` explainability traces with `traceType: "spl_match"` and `data` containing: `{ matchedFields: [...], unmatchedFields: [...], commandChain: parseSplPipeChain(source).map(c => c.command) }`.
   - Include `sourceLineHints` from `findSplSourceLineHints()`.
   - Compute summary (totalCases, passed, failed, matched, missed, falsePositives) same as sigma adapter.
   - Set engine to "client".

6. `buildExplainability(run)` -- Return `run.explainability` (same as sigma adapter).

7. `buildPublication(request)` -- Support two targets:
   - `"spl"`: Output the raw SPL source text as-is. Set converter id to "splunk-spl-identity", version "1.0.0".
   - `"json_export"`: Wrap in JSON with `_meta` (converter, exportedAt), `rule` (parsed pipe chain commands, original source, technique hints extracted from comments). Set converter id to "spl-to-json", version "1.0.0".
   - Throw Error for unsupported targets: `Unsupported SPL publish target "${target}"`.
   - Compute sourceHash and outputHash using `sha256Hex()` (same helper as sigma adapter -- copy the same async function).
