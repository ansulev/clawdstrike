**Main component `SplVisualPanel`** accepts `DetectionVisualPanelProps`:

1. **Header section** -- Parse comment lines from source (lines starting with `//`). Extract title, author, description from comment metadata. Show editable TextInput for title and TextArea for description. When edited, reconstruct the comment block and update source via onSourceChange.

2. **Pipe Chain section** -- Use `parseSplPipeChain(source)` to get `SplCommand[]`. Render each command as a **card** in a vertical pipeline:
   - Each card has a left border colored with accentColor (or DEFAULT_ACCENT for subcomponents).
   - Card header shows the command name in uppercase bold monospace (e.g. "SEARCH", "WHERE", "STATS", "TABLE").
   - Card body shows the args text.
   - Between cards, render a pipe connector (a vertical line with a `|` character, styled with opacity-50).

3. **Editable command cards** -- For "search" and "where" commands specifically:
   - Parse the args to extract `field=value` pairs.
   - Render each field-value pair as an inline editable row: field name label + TextInput for the value.
   - When user edits a value, reconstruct the command's rawText with the updated value and rebuild the full SPL source by joining all commands with ` | ` (preserving the original command structure for non-edited commands).
   - Call `onSourceChange(reconstructedSource)` to round-trip changes back to the text editor.
   - For other command types (stats, table, fields, etc.), show the args as read-only monospace text.

4. **Round-trip reconstruction** -- Build a helper `reconstructSplFromCommands(commands: SplCommand[], commentBlock: string): string`:
   - Start with the comment block (if any).
   - Join commands: first command's rawText, then each subsequent command prefixed with `\n| `.
   - This ensures editing a single field value in a card only changes that field, not the entire document.

5. **Read-only mode** -- When `readOnly` is true, render TextInputs as disabled/non-interactive.

6. **Empty state** -- If source is empty or parsing yields 0 commands, show a muted message: "Enter SPL query to see the pipe chain visualization."

**Part A: Create spl-translation-provider.ts**

Implement a `TranslationProvider` that handles two translation pairs:
1. `sigma_rule` -> `splunk_spl` (Sigma to SPL)
2. `splunk_spl` -> `sigma_rule` (SPL to Sigma)

```typescript
import type { TranslationProvider, TranslationRequest, TranslationResult, FieldMapping } from "./shared-types";
import { registerTranslationProvider } from "./translations";
import { convertSigmaToQuery } from "./sigma-conversion";
import { parseSplFieldConditions, parseSplPipeChain } from "./spl-parser";
import { getAllFieldMappings, type FieldMappingEntry } from "./field-mappings";
```

**`canTranslate(from, to)`:**
Return true for `("sigma_rule", "splunk_spl")` and `("splunk_spl", "sigma_rule")`.

**`translate(request)` -- sigma_rule -> splunk_spl direction:**
- Call `convertSigmaToQuery(request.source, "spl")` from sigma-conversion.ts (this already generates SPL from Sigma).
- Map the result: `success` from conversion result, `output` from conversion output.
- Build `fieldMappings` array: for each field used in the output, look up the field mapping entry and record `{ sigmaField, targetField: entry.splunkCIM, confidence }` where confidence is "exact" if `splunkCIM` is defined, "approximate" otherwise.
- Set `untranslatableFeatures` to empty array (single-event Sigma -> SPL is generally lossless for field conditions).
- Map diagnostics from `SigmaConversionDiagnostic[]` to `TranslationDiagnostic[]` (same shape).

**`translate(request)` -- splunk_spl -> sigma_rule direction:**
- Parse SPL conditions using `parseSplFieldConditions(request.source)`.
- Parse pipe chain using `parseSplPipeChain(request.source)` to extract index/sourcetype for logsource mapping.
- Build reverse field mapping: for each condition, find the `FieldMappingEntry` where `splunkCIM` matches the condition's field name. Use the `sigmaField` as the Sigma field name. If no mapping found, use the SPL field name as-is and add a diagnostic warning.
- Build logsource: map `sourcetype` to Sigma logsource category:
  - `WinEventLog:Security` or `WinEventLog:Sysmon` -> `{ category: "process_creation", product: "windows" }`
  - `firewall` -> `{ category: "network_connection" }`
  - default -> `{ category: "process_creation" }`
- Build detection block: create a `selection` object with mapped Sigma field names + modifiers:
  - modifier=null (exact) -> `fieldName: value`
  - modifier="contains" -> `fieldName|contains: value`
  - modifier="startswith" -> `fieldName|startswith: value`
  - modifier="endswith" -> `fieldName|endswith: value`
  - modifier="regex" -> `fieldName|re: value`
- Extract metadata from comment lines (title, author, description).
- Construct Sigma YAML using YAML.stringify():
  ```yaml
  title: <from comments or "Converted SPL Rule">
  status: experimental
  description: "Converted from SPL rule"
  author: <from comments or "Detection Lab">
  date: <today YYYY/MM/DD>
  logsource: <mapped>
  detection:
    selection: <mapped fields>
    condition: selection
  falsepositives: ["Review converted rule"]
  level: medium
  ```
- Build `fieldMappings` array showing each CIM->Sigma mapping used, with confidence.
- Populate `untranslatableFeatures` for SPL features that have no Sigma equivalent:
  - If pipe chain includes "stats", "eval", "lookup", "subsearch" commands, add them.
  - If conditions include `IN(...)` lists, note "Multiple-value IN clauses simplified".
- Return TranslationResult.
