Create `kql-adapter.ts` following the sigma-adapter.ts pattern. This file has four sections:

**Section 1: KQL Parser Utilities**

Create exported helper functions for parsing KQL text (used by this adapter and later by the visual panel):

- `parseKqlQuery(source: string): KqlParsedQuery` -- Extracts table name, where clauses, project columns, and extend expressions from KQL text. KQL structure: `TableName | where ... | where ... | project ... | extend ...`. Parse by splitting on `\n| ` (pipe at start of line) and classifying each segment by its operator keyword. Return:
  ```typescript
  export interface KqlWhereClause { field: string; operator: string; value: string; raw: string; }
  export interface KqlParsedQuery {
    tableName: string;
    whereClauses: KqlWhereClause[];
    projectColumns: string[];
    extendExpressions: string[];
    rawLines: string[];
    comments: string[];
  }
  ```
  - Where clause operator patterns: `==`, `!=`, `contains`, `!contains`, `startswith`, `!startswith`, `endswith`, `!endswith`, `matches regex`, `has`, `!has`, `in`, `!in`.
  - Parse where clause fields: split on operator, trim. Handle both `field operator "value"` and `field operator value`.

- `extractKqlWhereFields(source: string): KqlWhereClause[]` -- convenience wrapper returning just where clauses.

- `clientSideKqlMatch(payload: Record<string, unknown>, whereClauses: KqlWhereClause[]): boolean` -- Match evidence payload against parsed KQL where-clauses. All clauses must match (AND logic). For each clause:
  - `==`: case-insensitive string equality
  - `!=`: case-insensitive string inequality
  - `contains`: case-insensitive substring check
  - `!contains`: negated contains
  - `startswith` / `endswith`: case-insensitive prefix/suffix
  - `has`: whole-word match (word boundary check)
  - `matches regex`: regex test
  - `in`: value is in a parenthesized list

**Section 2: File Type Registration**

Call `registerFileType()` at module load with:
- `id`: `"kql_rule"`
- `label`: `"Microsoft Sentinel KQL Rule"`
- `shortLabel`: `"KQL"`
- `extensions`: `[".kql"]`
- `iconColor`: `"#0078d4"` (Microsoft blue)
- `testable`: `true`
- `convertibleTo`: `["sigma_rule"]`
- `defaultContent`: A valid KQL starter template:
  ```
  // KQL Detection Rule
  // Table: SecurityEvent
  // Description: Detects suspicious process creation
  SecurityEvent
  | where EventID == 4688
  | where CommandLine contains "powershell"
  | project TimeGenerated, Computer, CommandLine, ParentProcessName
  ```
- `detect` callback: return true if content matches KQL patterns -- check for table name prefix followed by pipe operators (`| where`, `| project`, `| extend`, `| summarize`). Avoid false positives with Sigma YAML or SPL.

**Section 3: KQL Adapter Implementation**

Create `const kqlAdapter: DetectionWorkflowAdapter` with `fileType: "kql_rule"`:

- **`canDraftFrom(seed)`**: Return true when `seed.dataSourceHints` includes any of `["process", "file", "network", "authentication"]` OR `seed.preferredFormats.includes("kql_rule")`.

- **`buildDraft(seed)`**: Generate valid KQL from seed data.
  1. Map `seed.dataSourceHints` to Sentinel table name:
     - "process"/"command" -> "SecurityEvent" (EventID 4688)
     - "file" -> "SecurityEvent" (EventID 4663)
     - "network" -> "CommonSecurityLog"
     - "authentication" -> "SigninLogs"
     - default -> "SecurityEvent"
  2. Map `seed.extractedFields` to Sentinel field names using `translateField(sigmaField, "sentinelField")` from field-mappings.ts. Fall back to the raw sigma field name if no mapping exists.
  3. Build KQL text: table name, then `| where` clauses for each field condition. Use `contains` for contains modifiers, `startswith`/`endswith` for those modifiers, `==` for exact match.
  4. Add `| project TimeGenerated, Computer` plus any mapped fields.
  5. Prepend comment header: `// {title}\n// Generated from {seed.kind} seed\n`.
  6. Return `{ source, fileType: "kql_rule", name, techniqueHints: seed.techniqueHints }`.

- **`buildStarterEvidence(seed, document)`**: Follow sigma-adapter pattern exactly. Create positive dataset from `seed.sourceEventIds` with `structured_event` items (normalize payload fields to Sentinel field names using `translateField`). Add one negative baseline item. Return EvidencePack.

- **`runLab(request)`**: Simulated client-side KQL execution.
  1. Get KQL source from `request.adapterRunConfig?.["kqlSource"]`.
  2. Parse where-clauses using `extractKqlWhereFields(source)`.
  3. For each evidence item (structured_event), run `clientSideKqlMatch(item.payload, whereClauses)`.
  4. Build `LabCaseResult` for each item (pass/fail based on expected vs actual match).
  5. Emit `plugin_trace` explainability traces with `traceType: "kql_where_match"` and `data` containing `{ tableName, matchedClauses: [...], unmatchedClauses: [...], matchedFields: [...] }`.
  6. Include `sourceLineHints` pointing to matching where-clause line numbers.
  7. Build LabRun summary with engine: "client".
  8. Return DetectionExecutionResult.

- **`buildExplainability(run)`**: Return `run.explainability` directly (same as sigma adapter).

- **`buildPublication(request)`**: Support two targets:
  1. `"kql"`: Output the raw KQL source text directly. Wrap in PublicationBuildResult with converter `{ id: "kql-identity", version: "1.0.0" }`.
  2. `"json_export"`: Wrap KQL in a Sentinel Analytics Rule JSON structure:
     ```json
     {
       "_meta": { "converter": "kql-to-analytics-rule", "converterVersion": "1.0.0", "exportedAt": "..." },
       "analyticsRule": {
         "displayName": "Detection Rule",
         "query": "<the KQL source>",
         "queryFrequency": "PT5H",
         "queryPeriod": "PT5H",
         "triggerOperator": "GreaterThan",
         "triggerThreshold": 0,
         "severity": "Medium",
         "tactics": [],
         "entityMappings": []
       }
     }
     ```
     Extract title from first comment line if present. Extract severity from seed hints or default to "Medium". Include MITRE tactics if parseable from comments.
  3. Throw Error for unsupported targets.

**Section 4: Auto-register**

Call `registerAdapter(kqlAdapter)` at module bottom. Export `kqlAdapter`.

**Part A: Create kql-translation.ts**

Create a `TranslationProvider` that handles two directions:

**Direction 1: sigma_rule -> kql_rule**
- Use `convertSigmaToQuery(source, "kql")` from sigma-conversion.ts.
- Map the `SigmaQueryConversionResult` to `TranslationResult`:
  - `success` = result.success
  - `output` = result.output
  - `diagnostics` = result.diagnostics mapped to TranslationDiagnostic
  - `fieldMappings`: For each Sigma field used in the query, look up its sentinelField via `getFieldMapping(sigmaField)`. Build FieldMapping[] with `{ sigmaField, targetField: sentinelField ?? sigmaField, confidence: sentinelField ? "exact" : "unmapped" }`.
  - `untranslatableFeatures`: `[]` (Sigma to KQL is generally lossless for single-rule translations)

**Direction 2: kql_rule -> sigma_rule**
- Parse the KQL source using `parseKqlQuery(source)` from kql-adapter.ts.
- Reverse-map Sentinel field names back to Sigma field names: iterate field mappings registry, find entries where `sentinelField` matches the KQL field, use that entry's `sigmaField`. If no reverse mapping found, use the KQL field name directly and add a warning diagnostic.
- Map KQL operators to Sigma modifiers:
  - `contains` -> `|contains`
  - `startswith` -> `|startswith`
  - `endswith` -> `|endswith`
  - `matches regex` -> `|re`
  - `==` / `has` -> no modifier (exact match)
- Map the KQL table name to Sigma logsource:
  - "SecurityEvent" -> `{ category: "process_creation", product: "windows" }`
  - "CommonSecurityLog" -> `{ category: "network_connection" }`
  - "SigninLogs" -> `{ category: "authentication", product: "azure" }`
  - "DeviceProcessEvents" -> `{ category: "process_creation", product: "windows" }`
  - "DeviceFileEvents" -> `{ category: "file_event", product: "windows" }`
  - "DeviceNetworkEvents" -> `{ category: "network_connection", product: "windows" }`
  - Default -> `{ category: "process_creation" }` with warning diagnostic
- Build Sigma YAML object with YAML.stringify():
  ```yaml
  title: "Converted from KQL"
  id: <crypto.randomUUID()>
  status: experimental
  description: "Auto-converted from KQL query targeting {tableName}"
  author: "Detection Lab"
  date: <today>
  logsource: <mapped logsource>
  detection:
    selection:
      <reversed field conditions>
    condition: selection
  falsepositives:
    - "Review required - auto-converted from KQL"
  level: medium
  ```
- Return TranslationResult with fieldMappings showing the reverse mapping and confidence levels.
- `untranslatableFeatures`: Populate with any KQL features that have no Sigma equivalent -- `| summarize`, `| join`, `| union`, `| extend` expressions, `ago()` time functions, `| project` (not semantically meaningful in Sigma). Add each as a string like `"summarize operator (aggregation not expressible in Sigma)"`.
