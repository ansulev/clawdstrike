    4. Fix `routeStatusBarItemContribution()` (SBAR-01). Replace the `render: () => null` placeholder with actual component resolution:
       ```typescript
       private routeStatusBarItemContribution(
         item: StatusBarItemContribution,
       ): Disposable {
         // Resolve the entrypoint to get the actual render component
         let resolvedComponent: ComponentType<any> | null = null;

         // Kick off async resolution — update the registration once resolved
         const resolvePromise = (async () => {
           try {
             const mod = await import(/* @vite-ignore */ item.entrypoint);
             resolvedComponent = mod.default ?? mod;
           } catch {
             // Entrypoint resolution failed — render stays as null gracefully
           }
         })();

         return statusBarRegistry.register({
           id: item.id,
           side: item.side,
           priority: item.priority,
           render: () => {
             if (resolvedComponent) {
               return createElement(resolvedComponent, { viewId: item.id });
             }
             return null;
           },
         });
       }
       ```

       NOTE: The status bar render function is called on every React render cycle, so once `resolvedComponent` is populated, the next render will show the real component. The initial null is acceptable for the brief async resolution window. The status bar's useSyncExternalStore does not need explicit notification since React's normal re-render cycle will pick up the resolved component.
