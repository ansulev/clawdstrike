<objective>
Build the dev console bottom panel tab that displays plugin lifecycle events, intercepted console output, contribution changes, and HMR timing information. The console provides real-time visibility into the plugin development loop.

Purpose: Plugin authors need to see what is happening during the edit-test-debug cycle -- when plugins activate, what contributions register, errors during loading, and console output from plugin code.
Output: A `PluginDevConsole` bottom panel component backed by a dev console event store and console interceptor.
</objective>

  <action>
    1. Create `apps/workbench/src/lib/plugins/dev/dev-console-store.ts`:
       - Uses the `useSyncExternalStore` pattern matching `status-bar-registry.ts` and `view-registry.ts`
       - Internal state: `events: DevLifecycleEvent[]` array (capped at 500 entries, FIFO)
       - `devConsoleStore` object with:
         - `push(event: DevLifecycleEvent): void` -- appends event, trims if > 500, notifies listeners
         - `getEvents(): DevLifecycleEvent[]` -- returns snapshot array
         - `clear(): void` -- empties events, notifies listeners
         - `subscribe(cb: () => void): () => void` -- standard subscribe for useSyncExternalStore
         - `getSnapshot(): DevLifecycleEvent[]` -- returns frozen snapshot for useSyncExternalStore
       - `useDevConsoleEvents(): DevLifecycleEvent[]` -- React hook using `useSyncExternalStore(devConsoleStore.subscribe, devConsoleStore.getSnapshot)`
       - `useDevConsoleFilter(pluginId?: string): DevLifecycleEvent[]` -- filtered version that returns events for a specific plugin
       - On module load, call `onDevLifecycleEvent((e) => devConsoleStore.push(e))` to wire up the HMR handler's lifecycle events into the store
       - Also subscribe to `pluginRegistry.subscribe('stateChanged', ...)` to emit lifecycle events for non-HMR state changes (activated, error, deactivated)

    2. Create `apps/workbench/src/lib/plugins/dev/console-interceptor.ts`:
       - `interceptConsole(pluginId: string): () => void`
         - Wraps `console.log`, `console.warn`, `console.error` with interceptors
         - Each interceptor calls the original console method AND pushes a DevLifecycleEvent to `devConsoleStore` with type `console:log|console:warn|console:error`
         - The message is formed by JSON.stringify-ing the arguments (or toString for simple values)
         - Returns a dispose function that restores original console methods
       - `stopIntercepting(): void` -- restores all original console methods (safety valve)
       - IMPORTANT: The interceptor must NOT create an infinite loop. It should NOT intercept its own console calls. Use a `isIntercepting` guard flag.
       - Scope: The interceptor is activated before a dev plugin's `activate()` is called and disposed after. It captures console output during plugin activation and runtime.

    3. Create `apps/workbench/src/lib/plugins/__tests__/dev-console-store.test.ts`:
       - Test push() adds events to the store
       - Test getEvents() returns all pushed events in order
       - Test clear() empties the events
       - Test cap at 500 entries (push 501, oldest is dropped)
       - Test useDevConsoleEvents hook (via renderHook) returns reactive snapshot
       - Test useDevConsoleFilter returns only events for specified pluginId

    4. Create `apps/workbench/src/lib/plugins/__tests__/console-interceptor.test.ts`:
       - Test interceptConsole wraps console.log/warn/error
       - Test intercepted calls still call original console methods
       - Test intercepted calls push DevLifecycleEvent to devConsoleStore
       - Test dispose restores original console methods
       - Test no infinite loop (interceptor's own console calls are not re-intercepted)

    5. Update `apps/workbench/src/lib/plugins/dev/index.ts` to also export:

       ```typescript
       export { devConsoleStore, useDevConsoleEvents, useDevConsoleFilter } from './dev-console-store';
       export { interceptConsole, stopIntercepting } from './console-interceptor';
       ```
