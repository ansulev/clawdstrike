  <action>
    Create Shodan plugin in `shodan-plugin.ts` following the same file structure as existing plugins in examples/:

    1. Define the plugin manifest with:
       - id: "clawdstrike.shodan"
       - name: "shodan-intel", displayName: "Shodan"
       - trust: "internal", categories: ["intel"]
       - activationEvents: ["onStartup"]
       - contributions.threatIntelSources: [{ id: "shodan", name: "Shodan", description: "Internet-connected device search engine", entrypoint: "..." }]
       - requiredSecrets: [{ key: "api_key", label: "Shodan API Key", description: "API key from account.shodan.io" }]

    2. Implement the ThreatIntelSource adapter:
       - supportedIndicatorTypes: ["ip", "domain"]
       - rateLimit: { requestsPerMinute: 60 } (1 req/sec)
       - enrich(indicator, context):
         - For IP: GET https://api.shodan.io/shodan/host/{ip}?key={apiKey}
         - For domain: GET https://api.shodan.io/dns/resolve?hostnames={domain}&key={apiKey}, then enrich the resolved IP
         - Map response to EnrichmentResult with:
           - verdict.classification: "suspicious" if vulns array non-empty, "benign" otherwise
           - verdict.confidence: scale by number of vulns (0 vulns = 0.3, 1-5 = 0.5, 6+ = 0.7)
           - verdict.summary: "X open ports, Y known vulnerabilities" or "No data found"
           - rawData: { ports, vulns, org, isp, country_name, city, os }
           - permalink: "https://www.shodan.io/host/{ip}"
           - cacheTtlMs: 3600000 (1 hour -- Shodan data changes slowly)

    3. Implement healthCheck():
       - GET https://api.shodan.io/api-info?key={apiKey}
       - Return true on 200

    4. Export shodanPlugin manifest and createShodanSource factory function.

    Auth pattern: Shodan uses `key` as a query parameter (NOT a header). This is per the Shodan API docs -- do NOT put the key in a header.
