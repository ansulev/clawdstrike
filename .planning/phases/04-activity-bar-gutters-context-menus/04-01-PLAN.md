**DesktopSidebar changes:**

1. Import `useViewsBySlot` from `@/lib/plugins/view-registry` and `ViewContainer` from `@/components/plugins/view-container`.

2. Add a `pluginNavSection` concept: after the 3 built-in navSections, render a "Plugins" section containing items from `useViewsBySlot("activityBarPanel")`. Group plugin items by their `meta.section` field. Each plugin item renders as a clickable button (NOT a Link, since it does not navigate to a route). Use the same visual styling as built-in NavItem links (icon, label, active state, collapsed layout).

3. The DesktopSidebar needs to communicate which plugin view is active. Add a module-level state store (simple useState + context, or a tiny zustand-like pattern) OR accept an `activePluginViewId` prop and an `onPluginViewSelect` callback. The simplest approach: use a shared state via a new lightweight `useActivePluginView` hook with `useSyncExternalStore`, similar to the ViewRegistry pattern. Create this hook inline in desktop-sidebar.tsx or in a small companion module. The state is: `activePluginViewId: string | null`. When a plugin item is clicked, set `activePluginViewId` to its id. When a built-in Link is clicked, set `activePluginViewId` to null. Export `useActivePluginView`, `setActivePluginView` for DesktopLayout to consume.

4. For plugin items in the sidebar: use a `<button>` element (not `<Link>`) styled identically to NavItem links. When clicked, call `setActivePluginView(viewId)`. The active state should use the same visual treatment (accent bar, background gradient, icon tint) as built-in active items.

5. For built-in NavItem links: wrap the existing `<Link>` onClick to also call `setActivePluginView(null)`, so clicking a built-in item clears any active plugin view. This ensures bidirectional switching.

6. The plugin section header should use accent color `"#6b8b55"` (a distinct green-tinted accent for the Plugins section) and title `"Plugins"`. Only render the section if there are plugin activity bar items registered.

7. For the icon rendering of plugin items: plugin items store an icon string (e.g. "IconShield"). Since the sidebar uses custom Sigil components (not Tabler icons directly), create a simple `PluginNavIcon` component that renders a small colored circle or the first letter of the label as a fallback. This avoids importing all possible Tabler icons dynamically. Style it to match the 15px size of Sigil icons.

**DesktopLayout changes:**

1. Import `useActivePluginView` from the sidebar module, `getView` from view-registry, and `ViewContainer` from view-container.

2. In the main content area rendering: check if `activePluginViewId` is non-null. If so, look up the ViewRegistration via `getView(activePluginViewId)` and render a `ViewContainer` with that registration instead of `<Outlet />`. Pass `isActive={true}`. Wrap the plugin view in the same `motion.div` animation wrapper as the Outlet for visual consistency.

3. If `activePluginViewId` is set but `getView()` returns undefined (plugin was uninstalled), automatically clear the active plugin view by calling `setActivePluginView(null)` in a useEffect.

4. The `<Outlet />` should still render when no plugin view is active (the default case for all built-in routes).

**ActivityBarPanelProps injection:**

Follow the same plugin-view-wrapper pattern from Phase 3 (BottomPanelPluginView/RightSidebarPluginView). In the DesktopLayout, when rendering a plugin activity bar panel, create a wrapper component (via useMemo) that clones the registration's component and injects `isCollapsed` from the sidebar state. This satisfies ABAR-03.

To get `isCollapsed` in DesktopLayout: either import the sidebar collapsed state from the workbench store (`state.ui.sidebarCollapsed` from `useWorkbench()`) or pass it through the active plugin view state. Since DesktopLayout already has access to the workbench store (it imports useMultiPolicy), and the sidebar collapsed state is in the same store, use `useWorkbench().state.ui.sidebarCollapsed`.
