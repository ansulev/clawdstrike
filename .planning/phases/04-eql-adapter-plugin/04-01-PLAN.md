Create `eql-parser.ts` with the following:

**AST types:**
```typescript
export type EqlEventCategory = "process" | "file" | "network" | "registry" | "dns" | "any";

export interface EqlCondition {
  field: string;        // ECS dotted path, e.g. "process.name"
  operator: "==" | "!=" | ":" | "~" | ">=" | "<=" | ">" | "<" | "in";
  value: string | string[];  // for "in" operator, value is string[]
  negated: boolean;
}

export interface EqlSingleQuery {
  type: "single";
  eventCategory: EqlEventCategory;
  conditions: EqlCondition[];
  logicOperator: "and" | "or";  // how conditions are joined
}

export interface EqlSequenceStep {
  eventCategory: EqlEventCategory;
  conditions: EqlCondition[];
  logicOperator: "and" | "or";
}

export interface EqlSequenceQuery {
  type: "sequence";
  byFields: string[];          // fields in "sequence by X, Y"
  steps: EqlSequenceStep[];
  maxspan?: string;            // e.g. "5m", "1h"
  until?: EqlSingleQuery;      // optional until clause
}

export type EqlAst = EqlSingleQuery | EqlSequenceQuery;

export interface EqlParseResult {
  ast: EqlAst | null;
  errors: string[];
}
```

**Parser function `parseEql(source: string): EqlParseResult`:**
- Trim and split on newlines; join continuation lines
- Detect sequence queries by checking if source starts with "sequence" (after stripping comments)
- For single-event queries: extract `{category} where {conditions}` pattern
  - Event category is the first word before "where"
  - Parse conditions: split on ` and ` / ` or ` (respect parentheses nesting)
  - For each condition, parse `field operator value` where:
    - `==` exact equals (case-sensitive)
    - `:` wildcard match (case-insensitive), e.g. `field : "*pattern*"`
    - `!=` not equals
    - `~` regex match
    - `>=`, `<=`, `>`, `<` numeric comparisons
    - `in` set membership, e.g. `field in ("a", "b")`
  - Handle quoted string values (double quotes) and wildcard patterns
  - Handle `not` prefix for negation
- For sequence queries: extract `sequence by {fields} [{step1}] [{step2}] ...`
  - Parse `by` clause field list (comma-separated)
  - Parse each `[{category} where {conditions}]` bracket block as a step
  - Parse optional `maxspan={duration}` after `by` clause or before first bracket
  - Parse optional `until [{category} where {conditions}]` at end
- Strip `//` comments from lines before parsing

**Generator function `generateEql(ast: EqlAst): string`:**
- For single queries: `{category} where {condition1} and {condition2}`
  - Format conditions: `field == "value"`, `field : "*pattern*"`, `field != "value"`, etc.
  - For `in` operator: `field in ("val1", "val2")`
- For sequence queries:
  ```
  sequence by {fields}
    [{maxspan clause if present}]
    [{category} where {conditions}]
    [{category} where {conditions}]
    [until {category} where {conditions}]
  ```
  - maxspan goes on the `sequence by` line: `sequence by host.id [maxspan=5m]`

**Helper function `extractEqlFields(ast: EqlAst): string[]`:**
- Return all unique field names referenced in conditions across all steps.

**Helper function `getEventCategoryForHint(hint: string): EqlEventCategory`:**
- Maps data source hints to EQL event categories:
  - "process" | "command" -> "process"
  - "file" -> "file"
  - "network" -> "network"
  - "registry" -> "registry"
  - "dns" -> "dns"
  - default -> "process"

Strip comment lines (starting with `//`) from input before parsing. If parse fails, return `{ ast: null, errors: ["..."] }` with descriptive error message.

Create `eql-adapter.ts` following the Sigma adapter pattern:

**File type registration at module load:**
```typescript
registerFileType({
  id: "eql_rule",
  label: "Elastic EQL Rule",
  shortLabel: "EQL",
  extensions: [".eql"],
  iconColor: "#f04e98",  // Elastic pink
  defaultContent: EQL_STARTER_TEMPLATE,
  testable: true,
  convertibleTo: ["sigma_rule"],
  detect: (filename: string, content: string) => {
    // Content-based detection for EQL syntax
    if (filename.endsWith(".eql")) return true;
    const trimmed = content.trim();
    // Check for event category prefix pattern: "process where", "file where", "network where", "sequence by"
    return /^(process|file|network|registry|dns|any)\s+where\b/i.test(trimmed)
      || /^sequence\s+by\b/i.test(trimmed);
  },
});
```

**EQL_STARTER_TEMPLATE constant:**
```
// Elastic EQL Detection Rule
// Event category prefix: process, file, network, registry, dns
process where process.name == "powershell.exe"
  and process.command_line : "*-encodedcommand*"
```

**Adapter implementation:**

`canDraftFrom(seed)`: Return true when `seed.dataSourceHints` includes any of "process", "command", "file", "network", "registry", or when `seed.preferredFormats` includes "eql_rule".

`buildDraft(seed)`:
- Determine event category from `seed.dataSourceHints` using `getEventCategoryForHint()`
- Map `seed.extractedFields` to ECS field names using `translateField(sigmaField, "ecsField")` from field-mappings.ts
- Build EQL conditions from extracted fields:
  - For "shell_command" actionType: `process.command_line : "*{target}*"`
  - For "file_access"/"file_write": `file.path : "*{target}*"`
  - For "network_egress": `destination.domain : "*{target}*"`
  - For generic targets: `process.command_line : "*{target}*"`
- If seed has multiple source events AND dataSourceHints suggest correlation, build a sequence query with multiple steps
- Generate EQL text via `generateEql(ast)`
- Return `{ source, fileType: "eql_rule", name: "Detection: ...", techniqueHints }`

`buildStarterEvidence(seed, document)`:
- Create structured_event evidence items from seed.sourceEventIds
- Normalize event payloads to include ECS field names (process.command_line, process.name, file.path, etc.)
- Add one negative baseline event
- Return EvidencePack with fileType "eql_rule"

`runLab(request)`:
- This is a stub for Plan 03 -- implement minimal version:
- Parse EQL source from `request.adapterRunConfig?.["eqlSource"]`
- Use parseEql() to get AST
- For each structured_event evidence item, do approximate client-side matching:
  - Extract ECS field values from event payload using dotted path lookup
  - For single queries: check all conditions against event fields
  - For sequence queries: check each step independently (approximate -- real sequence matching is in Plan 03)
- Emit `plugin_trace` ExplainabilityTrace with traceType "eql_match"
- Return DetectionExecutionResult with summary stats

`buildExplainability(run)`: Return `run.explainability` (pass-through like Sigma adapter).

`buildPublication(request)`:
- For target "eql": output raw EQL text with a comment header including rule title/author/severity
- For target "json_export": wrap in NDJSON detection rule format:
  ```json
  {
    "_meta": { "converter": "eql-export", "converterVersion": "1.0.0", "exportedAt": "..." },
    "rule": {
      "type": "eql",
      "language": "eql",
      "query": "{eql_source}",
      "name": "{derived from source or request}",
      "risk_score": 50,
      "severity": "medium",
      "tags": ["...mitre tags..."],
      "threat": [{ "framework": "MITRE ATT&CK", "technique": [...] }]
    }
  }
  ```
- Other targets: throw `Unsupported EQL publish target "${target}"`
- Use sha256Hex helper (same as sigma-adapter) for sourceHash/outputHash
