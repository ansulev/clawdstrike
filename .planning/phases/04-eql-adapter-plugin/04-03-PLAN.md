Create `eql-translation.ts` implementing a `TranslationProvider` with two translation directions:

**Direction 1: Sigma -> EQL (`from: "sigma_rule", to: "eql_rule"`):**

1. Parse Sigma YAML via `parseSigmaYaml(source)` -- if fails, return error diagnostics
2. Map Sigma `logsource.category` to EQL event category:
   - `process_creation` | `process_access` -> `"process"`
   - `file_event` | `file_access` | `file_change` -> `"file"`
   - `network_connection` -> `"network"`
   - `registry_add` | `registry_set` | `registry_event` | `registry_delete` -> `"registry"`
   - `dns` | `dns_query` -> `"dns"`
   - default -> `"process"` with warning diagnostic
3. Extract selection blocks from `detection` (same pattern as sigma-conversion.ts `extractSelections()`)
4. For each field in selection blocks:
   - Parse Sigma field key via `parseFieldKey(key)` -> `{ field, modifiers }`
   - Look up ECS field name via `translateField(field, "ecsField")`
   - If no mapping found, use the Sigma field name as-is and record as "unmapped" in fieldMappings
   - If mapping found, record as "exact" in fieldMappings
   - Map Sigma modifiers to EQL operators:
     - `contains` -> `:` operator with `"*{value}*"` wrapping
     - `startswith` -> `:` operator with `"{value}*"` wrapping
     - `endswith` -> `:` operator with `"*{value}"` wrapping
     - `re` -> `~` operator
     - no modifier -> `==` operator (exact match)
   - Handle multi-value fields: if Sigma field has array values, combine with `or`
5. Build EqlSingleQuery AST from conditions
6. Generate EQL text via `generateEql(ast)`
7. Add comment header: `// Translated from Sigma: {title}\n// NOTE: Review ECS field names for your Elastic deployment\n`
8. Return TranslationResult with output, diagnostics, fieldMappings, untranslatableFeatures: []

**Direction 2: EQL -> Sigma (`from: "eql_rule", to: "sigma_rule"`):**

1. Parse EQL via `parseEql(source)` -- if fails, return error diagnostics
2. Determine Sigma logsource from event category:
   - `"process"` -> `{ category: "process_creation", product: "windows" }`
   - `"file"` -> `{ category: "file_event" }`
   - `"network"` -> `{ category: "network_connection" }`
   - `"registry"` -> `{ category: "registry_event" }`
   - `"dns"` -> `{ category: "dns_query" }`
   - `"any"` -> `{ category: "process_creation" }` with warning
3. For single-event queries:
   - Reverse-map ECS field names to Sigma fields: iterate `getAllFieldMappings()`, find entries where `ecsField` matches, use `sigmaField`
   - If no reverse mapping, use the ECS field name as-is with warning diagnostic
   - Map EQL operators to Sigma modifiers:
     - `==` (exact) -> no modifier
     - `:` with `"*val*"` pattern -> `|contains` modifier, strip `*` from value
     - `:` with `"val*"` pattern -> `|startswith` modifier
     - `:` with `"*val"` pattern -> `|endswith` modifier
     - `~` (regex) -> `|re` modifier
     - `!=` -> prefix condition key with `not_` (or use separate filter block)
   - Build Sigma YAML with detection.selection block and condition: "selection"
4. For sequence queries:
   - Translate ONLY the first step's conditions into Sigma (Sigma cannot represent sequences)
   - Populate `untranslatableFeatures` array:
     - `"sequence correlation with {N} events not preserved"`
     - `"sequence by {fields} join not preserved"` if byFields present
     - `"maxspan={value} temporal constraint not preserved"` if maxspan present
     - `"until clause not preserved"` if until present
   - Add warning diagnostic about sequence -> single-event degradation
   - Use the first step's event category for logsource mapping
5. Build Sigma YAML:
   ```yaml
   title: "EQL Translation"
   status: experimental
   description: "Translated from EQL query"
   logsource: { ... }
   detection:
     selection:
       {field|modifier}: {value}
     condition: selection
   level: medium
   ```
6. Return TranslationResult with output, diagnostics, fieldMappings, untranslatableFeatures

Replace the stub `runLab` in `eql-adapter.ts` with a full client-side matching implementation:

**EQL condition evaluator function `evaluateEqlCondition(condition: EqlCondition, payload: Record<string, unknown>): boolean`:**
- Resolve dotted field path from payload (e.g. "process.command_line" -> payload.process?.command_line)
  - Use a helper `getNestedValue(obj, path)` that splits on "." and traverses
- If field not found in payload, return false (unless negated)
- Convert both sides to strings for comparison
- Switch on operator:
  - `==`: case-sensitive string equality
  - `!=`: case-sensitive string inequality
  - `:`: wildcard match (case-insensitive). Convert `*` to regex `.*`, anchor with `^...$`
  - `~`: regex match using `new RegExp(value, "i")`
  - `>=`, `<=`, `>`, `<`: numeric comparison (parseFloat both sides)
  - `in`: check if eventValue is in the value array (case-insensitive)
- Apply negation if `condition.negated` is true (invert result)

**Single-event matching function `matchSingleQuery(query: EqlSingleQuery, payload: Record<string, unknown>): { matched: boolean; matchedFields: string[]; unmatchedFields: string[] }`:**
- Check event category: if payload has an `event.category` field, verify it matches the query's eventCategory (skip for "any")
- Evaluate all conditions:
  - For "and" logic: all conditions must pass
  - For "or" logic: at least one condition must pass
- Track which conditions matched and which did not
- Return matched boolean + lists of matched/unmatched field names

**Sequence matching function `matchSequenceQuery(query: EqlSequenceQuery, events: Array<{ item: EvidenceItem; index: number }>): { perStepResults: Array<{ stepIndex: number; matchedEventIndices: number[]; matchedFields: string[][] }>; overallMatched: boolean }`:**
- For each step in the sequence, find events whose payload matches the step's conditions
- For simplicity (client-side approximation), each step is matched independently against all events
- A sequence "matches" if every step found at least one matching event
- Per-step results track which events matched which step and which fields

**Updated `runLab(request)` implementation:**
1. Extract EQL source from `request.adapterRunConfig?.["eqlSource"]` or `request.document` context
2. Parse via `parseEql(source)` -- if parse fails, return error result
3. Collect all structured_event evidence items across datasets
4. For single queries:
   - For each evidence item, call `matchSingleQuery(query, item.payload)`
   - Build LabCaseResult (pass/fail based on expected vs actual)
   - Build `plugin_trace` ExplainabilityTrace with:
     - `traceType: "eql_match"`
     - `data: { eventCategory, matchedFields, unmatchedFields, queryType: "single" }`
     - `sourceLineHints`: line numbers of matched condition fields in source
5. For sequence queries:
   - Call `matchSequenceQuery(query, allEvents)`
   - For each evidence item, determine which sequence step(s) it matched
   - Build LabCaseResult for each evidence item
   - Build `plugin_trace` ExplainabilityTrace with:
     - `traceType: "eql_sequence_match"`
     - `data: { stepIndex, totalSteps, stepCategory, matchedFields, unmatchedFields, sequenceComplete, queryType: "sequence" }`
     - `sourceLineHints`: line numbers of the matched step's bracket block
6. Compute summary: totalCases, passed, failed, matched, missed, falsePositives
7. Build ReportArtifact summary with engine: "client"
8. Return `{ run, coverage: null, reportArtifacts }`

**Helper `findEqlSourceLineHints(source: string, fields: string[]): number[]`:**
- Search source lines for occurrences of the field names
- Return matching 1-indexed line numbers
