**1. YARA-L Parser Helpers:**

Create `parseYaralRule(source: string)` that extracts structured data from YARA-L text:
- Returns `{ meta: Record<string, string>, events: YaralEventPredicate[], condition: string, ruleName: string } | null`
- Parse `rule NAME {` to extract ruleName
- Parse `meta:` section: key = "value" or key = value pairs
- Parse `events:` section: each line like `$varName.field.path = "value"` or `$varName.field.path = /regex/ nocase`
- Parse `condition:` section: the condition expression (e.g., `$e` or `$e1 and $e2`)
- This is a best-effort regex parser for client-side simulation, not a full grammar

Create `YaralEventPredicate` type:
```typescript
interface YaralEventPredicate {
  variable: string;    // e.g. "$e" or "$e1"
  fieldPath: string;   // e.g. "metadata.event_type" or "target.process.command_line"
  operator: "=" | "!=" | ">" | "<" | ">=" | "<=";
  value: string;       // the literal value or regex pattern
  isRegex: boolean;    // true if value was /pattern/
  nocase: boolean;     // true if "nocase" modifier present
}
```

Create `matchPredicateAgainstEvidence(predicate: YaralEventPredicate, payload: Record<string, unknown>): boolean`:
- Resolve `fieldPath` against the payload using dot-notation traversal (e.g., `target.process.command_line` traverses `payload.target.process.command_line`)
- Also try flat field lookup (e.g., `payload["CommandLine"]` with field mapping via `sigmaToUdm` inverse)
- For regex predicates: test the value against the regex pattern, respecting nocase
- For equality predicates: case-insensitive string comparison if nocase, exact otherwise
- Return true if the evidence field matches

**2. runLab Implementation (YARAL-03):**

Follow the sigma-adapter pattern:
- Extract source from `request.adapterRunConfig?.["yaralSource"]` as string
- If no source, return empty run with "No YARA-L source provided" report
- Parse the rule with `parseYaralRule(source)`
- Collect all evidence items from all datasets in the evidence pack
- Filter to structured_event and ocsf_event items
- For each event item:
  - Group predicates by variable name (e.g., all `$e` predicates together)
  - For each variable group, check if ALL predicates match (AND semantics within a variable)
  - The condition section determines how variable matches combine (for now, treat single-variable rules as "all predicates must match", multi-variable as "at least one variable's predicates match")
  - Create LabCaseResult with pass/fail based on expected vs actual
  - Create ExplainabilityTrace with `kind: "plugin_trace"`, `traceType: "yaral_match"`, `data` containing:
    - `matchedVariables`: array of `{ variable, matchedPredicates: [{ fieldPath, value }] }`
    - `unmatchedPredicates`: array of `{ fieldPath, expectedValue, actualValue }`
    - `ruleName`: from parsed rule
  - Include `sourceLineHints` by finding line numbers in source where matched field paths appear
- Compute summary: totalCases, passed, failed, matched, missed, falsePositives
- Return DetectionExecutionResult with the LabRun and a summary ReportArtifact

**3. buildPublication Implementation (YARAL-04):**

- `"yaral"`: Output is the raw source text. Manifest converter id `"yaral-identity"`, version `"1.0.0"`.
- `"json_export"`: Wrap the rule in a Chronicle-style JSON envelope:

  ```json
    "ruleName": "<parsed from source>",
    "ruleText": "<raw source>",

      "author": "<from meta>",
      "description": "<from meta>",
      "mitreTactics": ["<from meta if present>"],
      "created": "<from meta>"
    }
  }
  ```
  Manifest converter id `"yaral-json-export"`, version `"1.0.0"`.
- For any other target: `throw new Error('Unsupported YARA-L publish target "${target}"')`
- Compute sourceHash and outputHash using the same `sha256Hex()` pattern from sigma-adapter
