**1. Parser (internal to this file):**

Create `parseYaralSource(source: string)` that returns:
```typescript
interface ParsedYaralRule {
  ruleName: string;
  meta: Array<{ key: string; value: string }>;
  eventVariables: Array<{
    name: string;  // e.g. "$e1"
    predicates: Array<{
      fieldPath: string;  // e.g. "metadata.event_type"
      operator: string;   // e.g. "="
      value: string;      // e.g. "PROCESS_LAUNCH" or /regex/
      isRegex: boolean;
      nocase: boolean;
      lineNumber: number;
    }>;
  }>;
  condition: string;
  matchSection: string | null;   // optional match: section content
  outcomeSection: string | null; // optional outcome: section content
}
```

Parse using regex-based extraction (same approach as sigma-visual-panel.tsx uses for YAML):
- Extract rule name from `rule NAME {`
- Extract meta section between `meta:` and the next section keyword (`events:`, `match:`, `outcome:`, `condition:`)
- Extract events section: group predicates by variable name (everything before the first `.`)
- Extract condition, match, and outcome sections similarly
- Use `useMemo` to memoize parsing

**2. Regenerator:**

Create `regenerateYaralSource(parsed: ParsedYaralRule): string` that produces valid YARA-L text from the parsed structure. This enables round-tripping: parse -> edit -> regenerate -> onSourceChange.

**3. Sub-components:**

**MetaSection:** Renders meta fields (author, description, severity, created, mitre_attack) as editable inputs using `Section`, `FieldLabel`, `TextInput`, `TextArea`, `SelectInput` from detection-panel-kit. Severity uses `SelectInput` with options: INFORMATIONAL, LOW, MEDIUM, HIGH, CRITICAL. Display a `SeverityBadge` next to the select. If mitre_attack present, render each technique with `AttackTagBadge`.

**EventVariableCard:** A card for one event variable (e.g. `$e1`). Shows:
- Header with variable name (e.g. "$e1") in monospace, colored with accentColor
- List of predicates, each showing: fieldPath, operator dropdown (=, !=, >, <), value input
- "Add predicate" button (not in readOnly mode)
- Predicates are editable: changing a value triggers regeneration

**ConditionEditor:** Shows the condition expression in a `TextInput`. For simple rules (`$e`), this is informational. For multi-event rules (`$e1 and $e2`), the user can edit the condition logic.

**MatchOutcomeSection:** Optional section shown only if match: or outcome: blocks exist. Renders as a `TextArea` for raw editing (these are less common and don't need full visual decomposition).

**4. Main component `YaralVisualPanel`:**

```typescript
export function YaralVisualPanel({ source, onSourceChange, readOnly, accentColor }: DetectionVisualPanelProps) {
```

- Parse source with `useMemo`
- If parse fails, show "Unable to parse YARA-L rule" message with accentColor border
- Layout using `ScrollArea` (import from `@/components/ui/scroll-area`):
  - Rule name as editable `TextInput` at top
  - MetaSection
  - Divider
  - "Events" heading with count badge
  - EventVariableCard for each variable
  - Divider
  - ConditionEditor
  - MatchOutcomeSection (if present)

  1. Clone the parsed structure
  2. Apply the edit
  3. Call `regenerateYaralSource()` to produce new text
  4. Call `onSourceChange(newText)`
