Modify `performSearch` in search-store.ts to cancel stale searches:

1. Add a module-level variable above the store creation:
   ```typescript
   let activeSearchController: AbortController | null = null;
   ```

2. At the top of `performSearch`, before `set({ loading: true })`:
   - If `activeSearchController` is not null, call `activeSearchController.abort()`
   - Create a new `AbortController` and assign to `activeSearchController`
   - Capture `const controller = activeSearchController`
   - Capture `const queryAtDispatch = query` (the query value read from `get()`)

3. After the `await searchInProjectNative(...)` resolves (line ~133, after `const result = ...`):
   - Check `if (controller.signal.aborted) return;` -- do NOT set results, do NOT set loading: false. Just return silently. A newer search is in flight.
   - Check `if (queryAtDispatch !== get().query) return;` -- staleness guard. The query changed while we were waiting. Discard results silently.

4. In the `catch` block (line ~163): add the same `controller.signal.aborted` check at the top of the catch block -- if aborted, return silently without setting error state.

5. In the empty-query early return (line ~120-130): also abort any active controller and set `activeSearchController = null` so that clearing the input cancels pending searches and clears the spinner.

6. After successfully setting results (line ~144-151) or in the empty-query path: set `activeSearchController = null` to clean up the reference.

Key: since `searchInProjectNative` is a Tauri IPC call that does not accept AbortSignal, the abort pattern is "ignore the result" rather than "cancel the network request". The staleness guard (`queryAtDispatch !== get().query`) is the primary defense; the AbortController is the belt-and-suspenders cleanup mechanism.
